The honest hack.

Most hacks make a system do something it shouldn't. A white hack makes it tell more truth — it scans your code for the places it lies about its own state.

node bin/whitehack.js scan . View on GitHub

MIT · dependency-free · no telemetry · source-readable

The Node CLI and imported scan() API are canonical. The linked browser playground is a preserved, client-only demo of the original eight rules and does not include the v0.6 protocol or wallet/crypto-awareness pack.

The small lies software tells about itself

The failed read that silently becomes 0. The cached value served as if it were live. The stale price oracle read as current. The token transfer whose failure is dropped on the floor. The score shown to a person with no way to ask why.

These usually aren't bugs in the ordinary sense — the code runs fine. It just isn't honest about its own state, and someone downstream trusts it anyway.

Honest about itself

An honesty tool that overstated its own certainty would be the first thing it ought to flag. So:

  • ✓ It uses heuristics — text patterns, not a full understanding of your program. A flagged line may be a false positive.
  • ✓ Every finding carries a confidence label; only confident findings affect the exit code, so heuristic noise never breaks a CI gate.
  • ✓ An empty result is not proof the code is honest. It surfaces common lies — it cannot prove their absence.
  • ✓ It is a fast, low-noise continuous check — not an audit replacement.
  • ✓ Its regex definitions produce known self-reflection noise; deterministic tests, not a self-scan count, are the release gate.
